## 1. 项目概述

### 1.1 定位

传统 SAST 静态扫描 + 代码图谱分析 + AI 大模型语义审计一体化安全审计平台。

### 1.2 核心目标

| 指标 | 目标 |
|------|------|
| 静态告警误报率降低 | 80%+ |
| 业务逻辑漏洞检出率提升 | 40% |
| 代码审计人效提升 | 60% |
| 漏洞运营 | 全自动化闭环（发现→定级→分配→修复→复测→归档） |

### 1.3 支持技术栈

| 类别 | 覆盖范围 |
|------|---------|
| 后端语言 | Java, Go, PHP, Python |
| 前端语言 | Vue/React/JS/TS |
| 依赖解析 | Maven, Gradle, NPM, Pip, Composer |
| 仓库源 | GitLab, Gitee, GitHub |

## 4. 数据模型

### 4.1 Finding 核心记录

```java
public record Finding(
    String id,
    String scanId, String projectId, String filePath,
    int lineStart, int lineEnd, String codeSnippet,
    String vulnType,              // SQL_INJECTION, XSS, 等
    Severity severity,             // CRITICAL, HIGH, MEDIUM, LOW
    String ruleId, String description,
    // --- 新增 AI 字段 (Sprint 1) ---
    String aiVerdict,              // "exploitable" | "false_positive" | "suspicious"
    double aiConfidence,           // 0.0 ~ 1.0
    String aiExplanation,          // AI 判断依据
    String aiGeneratedPatch        // AI 修复补丁 (可选)
) {}
```

### 4.2 CPG 图数据模型 (Neo4j)

**Method 节点：**
```cypher
(:Method {
    key, name, className, signatureKey,
    returnType, annotations, startLine, endLine,
    isStatic, isPublic, projectId, scanId, isLatest
})
```

**CALLS 关系：**
```cypher
(:Method)-[:CALLS {lineNumber, callSignature, scanId}]->(:Method)
```

### 4.3 CodeX 请求/响应模型

```java
CodexRequest {
    scanId, vulnId, language, codeSnippet, filePath,
    lineStart, lineEnd, callChain, dataSource,
    reachable (boolean), timeout, extra (Map)
}

CodexResponse<T> {
    success (boolean), data (T), errorMessage,
    duration, modelVersion, fallbackLevel (enum)
}

FallbackLevel: NONE | API_FALLBACK | FULL_FALLBACK
```

## 6. API 集成矩阵

| 从 \ 到 | Engine | Codex-Integration | Frontend |
|---------|--------|-------------------|----------|
| Engine CPG | — | 结构化代码片段 + 调用链 | — |
| Codex Adapter | `analyzeVuln()` → aiVerdict | — | REST + WS |
| 前端 AI 面板 | — | `POST /api/v1/ai/analyze` | — |
| 前端图谱 | — | CPG JSON 格式 | @vue-flow 渲染 |
| 扫描状态 | — | — | WebSocket 4 主题 |

### 6.1 数据流（完整闭环）

```
1. 开发提交 → WebHook 触发
2. 调度中心 → 拉起 Docker 沙箱
3. 沙箱拉取代码 → SAST 扫描 + CPG 解析
4. 提取可疑漏洞片段、污点链路
5. RAG 召回历史漏洞与规范
6. CodeX Agent 主导审计 → 误报过滤 + 逻辑漏洞识别 + 风险定级
7. CodeX → POC 生成 + 修复补丁（V1.1）
8. → 漏洞工单 → 责任人 → 人工终审
9. 新漏洞样本入库，迭代更新规则
```

## 7. 核心设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| AI 架构 | CodeX Agent 主控 + LLM 辅助 | 解决通用大模型代码推理不准、补丁不可编译 |
| CPG 持久化 | Neo4j Java Driver（非 Spring Data Neo4j） | Engine 是 CLI 无 Spring Boot，轻量 |
| 前端图渲染 | @vue-flow + dagre | Vue 生态、DAG 原生支持、虚拟化 1000 节点 |
| 前端代码查看 | CodeMirror 6 | 轻量（Monaco 1/5）、虚拟滚动、多语言 |
| 实时推送 | STOMP over SockJS | Spring Boot 原生支持、多主题 |
| 大模型集成 | API 模式（远程调用） | 无需自管 GPU 基础设施 |
| 双模型分工 | CodeX = 代码任务 / LLM = 语义任务 | 差异化互补，精度提升 |
| aiVerdict | 三值：exploitable / false_positive / suspicious | suspicious 中间态需人工二审 |
